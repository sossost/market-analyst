# 일간 브리핑 수치 기준 명시 강제 (#158)

## 선행 맥락

**관련 이력:**

- **PR #82 (report-consistency)**: `pctFromLow52w` 오해석 방지를 위해 `systemPrompt.ts` 규칙 섹션에 "이 수치를 리포트에 인용할 때 반드시 '52주 저점 대비 +XX%'로 표기하세요" 규칙을 이미 추가함. 그러나 이 규칙은 `pctFromLow52w` 필드에만 한정되어 있고, LLM이 일관되게 준수하지 않음.

- **PR #63 (report-reliability)**: bull-bias 필터 3개를 시스템 프롬프트에 추가. "절차 강제" 형태(금지 + 정량 근거 요구)가 단순 "금지" 규칙보다 효과적임을 검증. 이번 수치 기준 강제도 동일 패턴 적용 가능.

- **PR #62 (reviewAgent.ts 정상화)**: 리뷰어 에이전트의 역할은 현재 5가지 관점(데이터 근거, 리스크, 실행 가능성, 편향, 정보 밀도)에만 집중. 수치 기준 누락 검증은 포함되어 있지 않음.

**핵심 교훈**: 규칙 추가만으로는 LLM 준수를 보장할 수 없다. 리뷰어 에이전트가 2차 방어선으로 기준 누락을 탐지해야 한다.

---

## 골 정렬

**ALIGNED** — 직접 기여.

Phase 2 주도섹터/주도주 포착의 핵심 목적은 독자가 투자 판단을 내리는 것이다. `AXTI(+105.7%)`처럼 기간과 기준이 없는 수치는 독자가 "지금 올라타도 되는가 아니면 이미 늦었는가"를 판단할 수 없게 한다.

- 52주 저점 대비 +105.7% = 이미 많이 오른 종목 (Phase 2 후기 가능)
- 5일 수익률 +7.9% = 이번 주 모멘텀 신호

같은 숫자라도 기준이 다르면 투자 판단이 반대가 된다. 기준 없는 수치는 알파가 아니라 노이즈다.

---

## 문제

일간 브리핑에서 퍼센트 수치를 기간/기준 없이 표시한다. 예: `AXTI(+105.7%)`.

현재 `systemPrompt.ts`에 "52주 저점 대비 +XX%로 표기하라" 규칙이 있으나, 이 규칙이 `pctFromLow52w` 필드에만 한정되어 있고 LLM이 다른 맥락의 수치(카탈리스트 언급, 기간 수익률 등)에는 적용하지 않는다. 또한 리뷰어 에이전트가 수치 기준 누락을 검증하지 않아 2차 방어선이 없다.

---

## Before → After

**Before**

- `systemPrompt.ts` 일간 프롬프트 규칙:
  - `pctFromLow52w`에 대해서만 "52주 저점 대비 +XX%로 표기" 규칙 존재
  - 다른 퍼센트 수치(일간 수익률, N일 수익률, 카탈리스트 언급 수치 등)에 대한 기준 명시 규칙 없음
- `reviewAgent.ts` REVIEWER_SYSTEM_PROMPT:
  - 5가지 검증 항목만 존재. "수치 기준 누락" 항목 없음
  - `AXTI(+105.7%)`처럼 기준 없는 수치가 있어도 OK 또는 REVISE 판정에 영향을 미치지 않음

**After**

- `systemPrompt.ts` 일간 프롬프트 규칙 강화:
  - 모든 퍼센트 수치에 기준 명시 의무화: `+X.X%(일간)`, `+X.X%(5일)`, `52주 저점 대비 +XX%`
  - 허용 기준 목록 명시: 일간, 5일, 20일, 52주 저점 대비
  - 기준 없는 수치(`AXTI(+105.7%)` 형태) 명시적 금지
- `reviewAgent.ts` REVIEWER_SYSTEM_PROMPT 강화:
  - 검증 항목 6번 추가: "수치 기준 명시 여부"
  - 퍼센트 수치에 기준이 누락된 경우 REVISE 판정 의무화

## 변경 사항

### 변경 1 — `src/agent/systemPrompt.ts`

**위치**: `buildDailySystemPrompt` 함수의 `## 규칙` 섹션

**현재 규칙 (관련 부분)**:
```
- **pctFromLow52w는 "52주 최저가 대비 현재 괴리율"입니다** — 종목의 최근 상승률이나 수익률이 아닙니다. 이 수치를 리포트에 인용할 때 반드시 "52주 저점 대비 +XX%"로 표기하세요. isExtremePctFromLow: true인 종목(주로 페니스탁)은 이 수치를 리포트에 노출하지 마세요.
```

**추가할 규칙**:
```
- **모든 퍼센트 수치에 기준을 명시하세요** — 기간이나 비교 기준이 없는 수치(예: `AXTI(+105.7%)`)는 독자가 의미를 해석할 수 없습니다. 허용 기준: `+X.X%(일간)`, `+X.X%(5일)`, `+X.X%(20일)`, `52주 저점 대비 +XX%`. 기준이 없으면 수치를 노출하지 마세요.
```

**주의**: 기존 `pctFromLow52w` 규칙과 병렬로 추가. 기존 규칙은 그대로 유지.

### 변경 2 — `src/agent/reviewAgent.ts`

**위치**: `REVIEWER_SYSTEM_PROMPT` 상수 (L40~L55)

**현재 검증 항목 (5개)**:
```
1. **데이터 근거**: 주장이 제시된 데이터로 뒷받침되는가?
2. **리스크 누락**: 빠진 위험 요인은 없는가?
3. **실행 가능성**: 제시된 종목에 실제 매수 근거가 구체적인가?
4. **편향 체크**: 불필요하게 낙관적이거나 비관적이지 않은가?
5. **정보 밀도**: 불필요한 장황함 없이 핵심만 전달하는가?
```

**추가할 항목 (6번)**:
```
6. **수치 기준 명시**: 퍼센트 수치에 기간/기준이 명시되어 있는가? `SYMBOL(+XX%)` 형태로 기준 없이 표기된 수치가 있으면 REVISE로 판정하세요. 허용 형태: `+XX%(일간)`, `+XX%(5일)`, `+XX%(20일)`, `52주 저점 대비 +XX%`.
```

## 테스트 계획

### systemPrompt.ts 테스트

`buildDailySystemPrompt()` 반환값에 아래 문자열이 포함되는지 검증:
- `"모든 퍼센트 수치에 기준을 명시하세요"` 또는 동등한 규칙 텍스트
- `"+X.X%(일간)"` 허용 형태 문자열

### reviewAgent.ts 테스트

리뷰어 판정 테스트:
- 입력: `AXTI(+105.7%)` 형태의 수치가 포함된 리포트 draft
- 기대: `verdict === "REVISE"` + `issues` 배열에 수치 기준 관련 항목 포함
- 입력: `+7.9%(5일)` 형태로 기준이 명시된 수치가 포함된 리포트
- 기대: 수치 기준만으로는 REVISE 판정되지 않음

**주의**: reviewAgent의 실제 LLM 호출은 mock 처리. `REVIEWER_SYSTEM_PROMPT` 변경 자체에 대한 유닛 테스트 (문자열 포함 여부)와, 기존 리뷰 파이프라인 통합 테스트(mock LLM)로 분리.

## 리스크

**프롬프트 규칙 vs LLM 실제 준수 격차**: 규칙을 추가해도 LLM이 일관되게 지키지 않는 것이 이 이슈의 원인이다. 이번 변경의 핵심은 리뷰어 에이전트를 2차 방어선으로 세우는 것 — 프롬프트 규칙(1차)이 실패해도 리뷰어(2차)가 잡도록 이중화한다.

**리뷰어의 LLM 판단 신뢰성**: 리뷰어 자체도 LLM이므로 수치 기준 누락 탐지가 완벽하지 않을 수 있다. 허용 형태를 구체적으로 열거(`+XX%(일간)`, `+XX%(5일)` 등)하여 패턴 매칭 수준의 명확한 기준을 제시하면 탐지 정확도가 높아진다.

**weeklySystemPrompt 미포함**: 이번 이슈 범위는 일간 브리핑에 한정. 주간 리포트에도 동일 문제가 존재한다면 별도 이슈로 처리.

## 의사결정 필요

없음 — 바로 구현 가능.

- 변경 범위: 일간 브리핑 한정 (`buildDailySystemPrompt` + `REVIEWER_SYSTEM_PROMPT`)
- 주간 프롬프트(`buildWeeklySystemPrompt`) 적용 여부: 이번 이슈 범위 밖으로 판단. 주간 리포트는 종목별 기간 데이터가 더 명시적으로 제공되어 상대적으로 문제 빈도가 낮음.
- 허용 기준 목록(일간/5일/20일/52주): 현재 시스템에서 실제 제공하는 기간 데이터에 기반한 목록. 확장 필요 시 별도 이슈.
