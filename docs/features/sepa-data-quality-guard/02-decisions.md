## DR-1: 방어 레이어를 ETL vs 스코어러 중 어디에 둘 것인가?

**Context:**
FMP 오염 데이터(누적 실적 혼입)를 ETL 단계에서 보정하거나, 스코어러 단계에서 감지하고 무력화하는 두 접근이 있다.

**Options Considered:**

### Option A: ETL 단계 보정
- **Pros:** 데이터가 처음부터 깨끗함. 하류 코드 변경 없음.
- **Cons:** FMP API의 누적/단독 혼재를 정확히 구분하는 것이 불가능에 가까움 (API 문서 없음, 기업별 회계 방식 다양). 오탐 시 정상 데이터를 훼손. ETL 로직이 과도하게 복잡해짐.
- **Effort:** L
- **Risk:** High

### Option B: 스코어러 레벨 방어 (채택)
- **Pros:** 원본 데이터 보존. 감지 실패 시에도 원본 조회 가능. 로직이 집중되어 테스트 가능. 의심 종목만 격리하므로 정상 종목에 영향 없음.
- **Cons:** 오염 데이터가 DB에는 남아 있음. 감지 로직의 임계값 튜닝 필요.
- **Effort:** M
- **Risk:** Low

**Decision:** Option B — 스코어러 레벨 방어.
ETL을 "정확성을 보장"하는 레이어로 사용하는 것은 현실적으로 불가능하다. 스코어러가 "신뢰할 수 있는 데이터인가?"를 판단하는 게이트키퍼 역할을 맡는 것이 더 안전하고 유지보수 가능하다.

## DR-2: 금융섹터를 SEPA 스코어링에서 제외할 것인가, 별도 기준을 적용할 것인가?

**Context:**
SEPA 기준(EPS/매출 YoY >25%, 마진 확대)은 성장주 중심 프레임워크다. 은행/보험은 EBITDA 무의미, ROE가 핵심 지표, 배당 수익률이 중요한 전혀 다른 평가 체계를 요구한다.

**Options Considered:**

### Option A: 금융섹터 완전 제외 → N/A
- **Pros:** 간단. 오염 위험 제거. 우리 시스템의 핵심 타깃(성장주)에서 금융주는 실제로 우선순위가 낮음.
- **Cons:** 금융주 중 Phase 2 진입 종목을 놓칠 수 있음.
- **Effort:** S
- **Risk:** Low

### Option B: 금융섹터 전용 스코어링 로직
- **Pros:** 금융주도 커버.
- **Cons:** ROE 데이터 미확보, 배당 데이터 미확보, 금융섹터별 세분화(은행/증권/보험) 필요. 지금 당장 구현할 준비가 안 됨.
- **Effort:** XL
- **Risk:** High

**Decision:** Option A — 금융섹터 N/A 제외.
프로젝트 골이 성장주 초입 포착이다. 금융주는 SEPA 기준이 원래 적용 대상이 아니다. 별도 로직은 데이터 인프라(ROE, 배당)가 갖춰진 후 별도 피처로 구현한다.

**대상 섹터 값:** `'Financial Services'`, `'Financials'`, `'Banks'`, `'Insurance'` (FMP symbols.sector 기준)

## DR-3: 일본 기업(비표준 회계연도)을 어떻게 식별할 것인가?

**Context:**
일본 기업은 FMP가 누적 실적을 단독 분기로 제공하는 빈도가 높다. 국가 코드(JP)로 일괄 제외하는 방법과, 데이터 이상 패턴 자체로 감지하는 방법이 있다.

**Options Considered:**

### Option A: 국가 코드 기반 제외 (country = 'JP')
- **Pros:** 명확. 구현 단순.
- **Cons:** 일본 기업 중 정상 데이터를 제공하는 종목도 존재할 수 있음. 일본만의 문제가 아님 (중국, 한국도 유사 이슈).
- **Effort:** S
- **Risk:** Medium (오탐 가능)

### Option B: 데이터 이상 패턴 감지 (채택)
- **Pros:** 국가에 무관하게 오염 데이터를 잡아냄. 일본 외 다른 나라 이슈도 커버. 정상 일본 기업은 패스. 임계값을 벗어난 실제 오염만 격리.
- **Cons:** 임계값 설정 필요. 엣지 케이스 존재.
- **Effort:** M
- **Risk:** Low

**Decision:** Option B — 패턴 감지.
국가 코드 제외는 과도하게 넓고, 문제의 본질은 "국가"가 아니라 "데이터 패턴"이다. QoQ 급변(5배)과 단위 불연속(10배) 두 가지 감지로 누적 혼입의 대부분을 커버할 수 있다.

## DR-4: 이상 데이터 감지 시 등급을 N/A로 할 것인가, F로 할 것인가?

**Context:**
감지된 이상 데이터 종목에 F를 부여하면 "실적이 나쁜 것"처럼 보인다. N/A는 "평가 불가"를 명확히 구분한다.

**Options Considered:**

### Option A: F 등급
- **Pros:** 타입 변경 없음. 기존 코드 흐름 최소 변경.
- **Cons:** 데이터 오염과 실적 부진을 구분할 수 없음. 리포트에서 "F등급(실적 미달)"로 잘못 해석될 수 있음. 추후 데이터가 정상화되어도 이력 추적 어려움.
- **Effort:** S
- **Risk:** Medium (의미 혼동)

### Option B: N/A 등급 (채택)
- **Pros:** 의미 명확 — "평가 불가 = 데이터 신뢰 불가". 리포트에서 별도 처리 가능. 이력 추적 가능 (왜 N/A인지 flags 기록).
- **Cons:** `FundamentalGrade` 타입 변경, 관련 switch문 업데이트 필요.
- **Effort:** S~M
- **Risk:** Low

**Decision:** Option B — N/A 등급.
F는 "실적이 기준 미달"이라는 의미다. 데이터를 신뢰할 수 없는 상황을 F로 묻어버리면 나중에 디버깅이 불가능해진다. N/A와 F를 명확히 분리하는 것이 장기적으로 올바른 설계다.

## DR-5: 기존에 이미 산출된 오염 등급은 어떻게 처리할 것인가?

**Context:**
`fundamental_scores` 테이블에 이미 SMFG를 포함한 오염 등급이 저장되어 있다.

**Options Considered:**

### Option A: 기존 레코드 그대로 두기 (canSkip 로직이 재스코어링 방지)
- **Pros:** 코드 변경 최소화.
- **Cons:** 새 실적이 없으면 기존 오염 스코어가 계속 사용됨. 주간 리포트에 오염 등급이 계속 노출될 수 있음.
- **Risk:** High

### Option B: 배포 시 마이그레이션 스크립트 실행 (채택)
- **Pros:** 즉시 효과. 기존 오염 등급 제거. `forceRescore: true` 옵션으로 전체 재스코어링 트리거 가능.
- **Cons:** 마이그레이션 스크립트 작성 필요.
- **Effort:** S
- **Risk:** Low

**Decision:** Option B — 마이그레이션 스크립트.
구현 완료 후 `scripts/migrate-invalidate-contaminated-scores.ts`를 한 번 실행하여 기존 스코어를 N/A로 갱신한다. 이후 정기 스코어링 시 새 방어 로직으로 재산출된다.

구체적 처리: 금융섹터 종목의 기존 스코어를 N/A로 업데이트. 데이터 이상 패턴 종목은 forceRescore로 재산출.

## DR-6: QoQ 급변 임계값을 몇 배로 설정할 것인가?

**Context:**
누적 실적이 단독 분기로 전환될 때의 패턴: 연간 4분기 누적이 1분기로 표기되면 약 4배 감소. 이를 감지하려면 임계값이 4배 이하여야 하나, 너무 낮으면 정상적인 계절성 변동(1분기 매출 급증 등)도 오탐된다.

**Decision:** 5배 (약 400% 변화).
- 4배 임계값: 계절성 강한 기업(예: 리테일 4분기 피크)에서 오탐 가능성
- 5배 임계값: 누적 혼입(~4배 변화)은 잡으면서, 진짜 폭발적 성장(5배 이상 QoQ)은 거의 없으므로 오탐 위험 낮음
- 양방향 감지: 급등(5배 이상)과 급감(1/5 이하) 모두 플래그
