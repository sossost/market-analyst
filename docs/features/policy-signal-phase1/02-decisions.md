# 의사결정 기록 — 정책-규제 선행 신호 추적 Phase 1

**날짜**: 2026-03-09
**상태**: accepted
**작성**: mission-planner

## 결정 1: Phase 1 범위를 프롬프트 레벨로 한정한다

### 컨텍스트

이슈 #95는 "정책-규제 선행 신호 추적"이라는 넓은 범위다. 구현 가능한 방법은 크게 세 가지다:

**Option A — 프롬프트 보강만 (Phase 1 채택)**
- 변경 파일: `geopolitics.md`, `newsCollector.ts`, `round2-crossfire.ts`
- DB 변경 없음. 별도 파이프라인 없음.
- 효과: 기존 토론 흐름 안에서 에이전트가 정책 신호를 체계적으로 분류

**Option B — DB에 정책 신호 테이블 추가**
- `policy_signals` 테이블 신설. `narrative_chains`의 `policy_signal_ids`와 연결.
- Wave 2b 스펙에서 Wave 3+로 명시적으로 연기된 항목.
- 비용: DB 마이그레이션 + ETL 로직 + 저장/조회 레이어 전체

**Option C — 외부 입법 데이터 API 연동**
- congress.gov API, GovTrack, Quorum 등 입법 진행 데이터 실시간 수집
- 비용: 데이터 모델링 + API 연동 + 정규화 레이어

### 결정: Option A

**근거:**

1. Wave 2b 스펙이 `policy_signal_ids` 필드를 DB에서 명시적으로 연기했다. 이를 뒤집으려면 체인 데이터가 5개 이상 축적된 후 교차 분석 패턴이 확인돼야 한다. 현재(2026-03-09)는 Wave 2b가 방금 머지(PR #101)된 시점으로, 데이터가 0건이다.

2. 정책 추적의 핵심 가치는 "입법 단계별 임팩트 확실성 차별화"인데, 이것은 분석 프레임의 문제지 데이터베이스의 문제가 아니다. LLM이 체계적인 평가 기준을 갖고 있지 않은 것이 현재 갭이므로, 프롬프트 보강이 가장 직접적인 해결책이다.

3. 외부 입법 API 연동(Option C)은 데이터 품질과 비용이 검증되지 않은 상태에서 인프라 의존성을 추가하는 것이다. LLM이 뉴스에서 단계를 판단하는 것이 먼저 작동하는지 확인한 뒤 결정해야 한다.

**수용하는 트레이드오프:**
- LLM이 뉴스에서 입법 단계를 해석하기 때문에 오분류 가능성이 있다.
- 정책 신호가 구조화된 DB에 저장되지 않아 히스토리 분석이 불가능하다.
- 이 두 가지는 Phase 2(DB 구현)에서 해결한다.

## 결정 2: 5단계 프레임을 채택하고 Stage 3을 임팩트 확실성 분기점으로 설정한다

### 컨텍스트

입법 진행 단계를 몇 단계로 나눌 것인가. 선택지:
- 3단계: 제안 → 확정 → 집행
- 5단계: 제안 → 심의 → 서명/발효 → 규정준비 → 집행
- 7단계 이상: 소위원회/본회의 등 세분화

### 결정: 5단계

**근거:**

입법 과정에서 투자 판단에 실질적인 의미가 있는 분기점이 5개다:
- 제안(S1)과 심의(S2)는 불확실성이 높아 "기대감"으로만 다룬다.
- 서명/발효(S3)가 투자 사이클 전환의 핵심 분기점이다. 기업들이 이 시점에 투자 계획을 실행으로 전환한다.
- 규정 준비(S4)는 S3 이후에도 실제 집행까지 수개월 지연이 발생하는 단계다. IRA 보조금이 대표 사례 — 법은 통과됐지만 신청 기준 확정까지 1년 이상 걸렸다.
- 실제 집행(S5)은 매출 반영 시점이지만 이미 시장 선반영이 끝난 경우가 많다.

3단계는 S4(규정준비)를 누락해서 "법 통과 = 즉시 집행"이라는 오해를 유발한다. 7단계 이상은 프롬프트 복잡도가 너무 높아지고 LLM이 일관된 분류를 유지하기 어렵다.

**관세/무역 정책의 5단계 매핑:**

관세 정책은 별도 섹션을 만들지 않고 동일한 5단계 프레임에 매핑한다:
- S1: 관세 검토/위협 발언 (Proposal)
- S2: 협상 진행 중 / 유예 기간 (Committee 수준의 불확실성)
- S3: 관세율 확정·발표 (Announced/Signed — 발효일 명시)
- S4: 발효 준비 기간 (기업 공급망 조정 시작)
- S5: 실제 부과 시작 / 보복 관세 발동

## 결정 3: 출력을 마크다운 테이블로 유지하고 JSON 파싱을 도입하지 않는다

### 컨텍스트

geopolitics 에이전트의 정책 신호 평가를 구조화하는 방법:

**Option A — 마크다운 테이블 (채택)**
- 기존 출력 형식과 일관성 유지
- moderator와 다른 에이전트들이 round1/2 텍스트를 읽을 때 자연스럽게 참조
- DB 저장 불필요

**Option B — JSON 블록 추가**
- market_regime이 하는 방식과 동일 (round3-synthesis.ts에서 파싱)
- Phase 2에서 DB 저장 연동 시 유리한 구조
- 현재 단계에서는 파싱 로직 + 유효성 검사 + 저장 코드가 필요해 과잉 복잡도

### 결정: Option A (마크다운 테이블)

**근거:**

Phase 1의 목표는 "LLM이 정책 신호를 체계적으로 평가하는 습관을 만드는 것"이다. 이 단계에서 JSON 파싱까지 구현하면 범위가 Phase 2까지 확장된다.

마크다운 테이블은 round2 교차 검증 때 다른 에이전트들이 텍스트로 읽고 반박하기에 충분하다. JSON이 필요해지는 시점은 이 데이터를 DB에 저장하고 히스토리를 분석할 때인데, 그것은 `policy_signal_ids` 구현(Wave 3+)에서 처리한다.

## 결정 4: round2-crossfire.ts에도 정책 단계 검증 항목을 추가한다

### 컨텍스트

Phase 1의 주요 리스크는 LLM이 입법 단계를 자의적으로 오분류하는 것이다. 이 리스크를 어떻게 완화할 것인가:

**Option A — geopolitics.md 변경만**
- 가장 적은 변경
- geopolitics 에이전트가 스스로 단계를 판단하고 오분류해도 교차 검증 없음

**Option B — round2-crossfire.ts에도 검증 항목 추가 (채택)**
- round2에서 macro, tech, sentiment 에이전트들이 geopolitics의 단계 판단을 명시적으로 검증
- 오분류 감지 가능성 증가
- 변경 범위: crossfire 프롬프트 4~6줄 추가

### 결정: Option B

**근거:**

시스템의 자기 교정 능력을 높이는 것이 프롬프트 엔지니어링의 핵심 원칙이다. geopolitics가 "Stage 3 완료"라고 잘못 판단했을 때, 교차 검증 없이 그 판단이 moderator 종합까지 그대로 전달되면 리포트 품질이 저하된다. round2에서 다른 에이전트들이 "뉴스를 보면 아직 협상 중이지 않은가?"를 지적할 수 있는 구조가 필요하다.

변경 비용(6줄)이 작고, 효과(자기 교정 회로)가 크다.

## 결정 5: newsCollector.ts에 추가할 쿼리를 2개로 제한한다

### 컨텍스트

뉴스 쿼리를 몇 개나 추가할 것인가:

**Option A — 2개 추가 (채택)**
- 의회 통과/서명(Stage 3) 포착용 1개
- 관세 발효일/집행(Stage 3~5) 포착용 1개

**Option B — 4~5개 추가**
- 각 단계별 전용 쿼리 설계 (Stage 1: 법안발의, Stage 2: 위원회통과, Stage 3: 서명, ...)
- 더 많은 커버리지, 더 많은 비용과 수집 시간

### 결정: Option A (2개)

**근거:**

뉴스 검색의 목적은 "오늘 어떤 정책 이벤트가 발생했는가"를 수집하는 것이다. 쿼리가 많아질수록 중복이 늘고, Brave API 호출 비용과 지연이 증가한다. 현재 geopolitics 쿼리 중 `"US legislation bill regulation tariff subsidy sector impact latest"`가 이미 일반적인 정책 뉴스를 커버한다.

추가가 필요한 것은 "단계 전환" 이벤트를 명시적으로 포착하는 쿼리다:
- "passed/signed/enacted" 키워드 → Stage 3 이벤트
- "effective date/implementation/enforcement" 키워드 → Stage 3~5 이벤트

2개면 충분하다. 더 필요하면 운영 관찰 후 추가한다.

## Phase 2 예약 결정

Phase 2에서 처리할 내용 (현재 결정 아님):

1. **`policy_signals` DB 테이블**: `narrative_chains` 체인 데이터 5개+ 축적 후 `policy_signal_ids` 연결 구현
2. **JSON 파싱 및 자동 저장**: geopolitics 출력의 정책 신호 평가를 DB에 구조화된 형태로 저장
3. **외부 입법 데이터 API 연동**: congress.gov 또는 유사 API를 통한 실시간 입법 진행 현황 수집
4. **레짐별 정책 임팩트 상관관계 분석**: `market_regimes`와 `policy_signals` 교차 분석

Phase 2 착수 기준: Phase 1 운영 관찰 3회 이상 + 정책 신호 테이블이 geopolitics 출력에서 일관되게 생성됨 확인.
