# 자율 이슈 처리 시스템

## 선행 맥락

- **launchd 인프라 완비** (PR #64): 맥미니에서 etl-daily, debate-daily, agent-weekly, qa-weekly 등 6개 잡 운영 중. 신규 잡 추가는 plist 1개 + setup-launchd.sh 배열 추가만으로 완료.
- **머지 권한 (ORGANIZATION.md)**: CEO 지시 없는 머지 절대 금지. 자율 처리 시스템도 PR 생성까지만. 머지는 CEO.

## 골 정렬

**SUPPORT** — 직접 주도주 포착 기능은 아니나, 매니저가 이슈 처리에 소비하는 인지 부하를 제거하여 전략적 판단에 집중할 수 있게 한다.

## 문제

CEO가 열린 이슈를 하나씩 매니저에게 전달해야 처리가 시작된다. CEO의 가용 시간에 이슈 처리 속도가 종속됨.

## Before → After

**Before**
- CEO가 이슈를 발견 → 매니저에게 전달 → 매니저가 처리 시작
- 이슈 처리 속도가 CEO의 가용 시간에 종속

**After**
- 매니저가 평일 09:00~18:00 KST 매 1시간마다 이슈를 자율 점검
- 자율 처리 가능한 이슈: 분석 → 구현 → 리뷰 → PR 생성까지 자동
- CEO 판단이 필요한 이슈: 이슈 코멘트로 에스컬레이션
- CEO는 PR만 확인하고 머지 결정만 한다

## 범위

### Phase 1 (MVP) — 이번 구현 대상

**핵심**: 이슈 트리아지 + 자율 실행 파이프라인. 트리아지부터 PR 생성까지 한 번에.

어차피 PR 리뷰에서 CEO가 최종 판단하므로, 잘못된 구현이 나와도 PR 리젝트하면 끝. 트리아지 검증 기간 없이 바로 자율 실행.

**포함:**
1. 스케줄러 스크립트 (`scripts/cron/issue-processor.sh`)
2. launchd plist (`com.market-analyst.issue-processor.plist`) — 평일 09:00~18:00 KST, 매 1시간
3. 이슈 트리아지 로직 (`src/issue-processor/triageIssue.ts`)
   - GitHub Issues API로 열린 이슈 조회
   - LLM 기반 자율 처리 가능 여부 판단
   - 이슈에 라벨 부착 + 코멘트 추가
4. 자율 실행 파이프라인 (`src/issue-processor/executeIssue.ts`)
   - `auto:queued` 이슈 → Claude Code CLI로 처리
   - 브랜치 생성 → 구현 → PR 생성
5. 에스컬레이션 코멘트 (CEO 판단 필요 이슈에 이유 명시)

**제외 (Phase 2):**
- Discord/Slack 알림
- 처리 결과 DB 저장 및 통계
- 일일 처리 건수 제한 자동 조절

## 라벨 상태 머신

```
[신규 이슈] (auto: 라벨 없음)
    │
    ▼ 트리아지 실행
    ├── 자율 처리 가능 → [auto:queued]
    │       │
    │       ▼ 파이프라인 실행
    │   [auto:in-progress]
    │       │
    │       ├── 성공 → [auto:done] + PR 링크 코멘트
    │       └── 실패 → [auto:needs-ceo] + 실패 사유 코멘트
    │
    └── CEO 판단 필요 → [auto:needs-ceo]
            │
            ▼ CEO가 지시 내리면
        라벨 제거 → 일반 미션 프로토콜
```

**라벨 정의:**

| 라벨 | 의미 | 자동/수동 |
|------|------|---------|
| `auto:queued` | 트리아지 완료, 처리 대기 | 자동 |
| `auto:in-progress` | 파이프라인 실행 중 | 자동 |
| `auto:done` | PR 생성 완료 | 자동 |
| `auto:needs-ceo` | CEO 판단 필요 | 자동 |

**중복 방지**: 이미 `auto:` 접두사 라벨이 붙은 이슈는 트리아지 스킵.

## 자율 처리 판단 기준

### 트리아지 방법

LLM(Claude)이 이슈 본문을 분석하여 판단. 규칙 기반보다 정밀하고, 어차피 잘못 판단해도 PR 리뷰에서 걸러짐.

### 자율 처리 가능 조건 (AND)

다음 **전부** 해당해야 자율 처리 가능:

1. **이슈 타입**: 버그 픽스, 명확한 개선, 리팩토링
2. **기존 패턴**: 코드베이스에 선례가 있는 수정
3. **의존성 없음**: 다른 미완료 이슈/PR에 블로킹되지 않음
4. **프로젝트 골 일치**: 현재 프로젝트 방향과 부합

### 자율 처리 불가 조건 (OR)

다음 **하나라도** 해당하면 `auto:needs-ceo`:

1. **새 기능**: 기존에 없는 기능 추가
2. **아키텍처 변경**: 모듈 구조, DB 스키마, API 계약 변경
3. **트레이드오프**: 명확한 정답이 없고 방향 선택이 필요
4. **외부 시스템 연동**: 새 API, 새 서드파티 서비스 추가
5. **불명확한 요구사항**: 이슈 본문만으로 구현 범위가 불분명
6. **비용 증가**: Claude API 호출 비용이 유의미하게 증가할 것으로 예상

**원칙: 판단이 애매하면 `auto:needs-ceo`.**

## 실행 파이프라인

`auto:queued` 이슈를 처리하는 순서:

```
1. 브랜치 생성 (fix/issue-{번호} 또는 feat/issue-{번호})
2. Claude Code CLI 실행 — 이슈 본문을 프롬프트로 전달
   - 기존 미션 프로토콜 준수 (mission-planner → 구현 → review)
3. 변경사항 커밋 + 푸시
4. PR 생성 (gh pr create)
5. 이슈에 PR 링크 코멘트
6. 라벨: auto:in-progress → auto:done
```

**실패 시:**
- 라벨: `auto:in-progress` → `auto:needs-ceo`
- 이슈에 실패 사유 코멘트
- 다음 이슈로 넘어감 (격리)

**1사이클 제한**: 한 번의 스케줄 실행에서 최대 2건 처리 (비용 + 시간 제한)

## 실패/에러 처리

| 시나리오 | 처리 |
|---------|------|
| GitHub API 인증 실패 | 로그 기록 + 다음 사이클 재시도 |
| 이슈 조회 실패 | 로그 기록 + 종료 |
| 라벨 부착 실패 | 로그 기록 + 코멘트만 달고 계속 |
| Claude Code CLI 실패 | `auto:needs-ceo` 부착 + 실패 사유 코멘트 |
| PR 생성 실패 | `auto:needs-ceo` 부착 + 실패 사유 코멘트 |
| 스크립트 예외 | stderr 로그 기록 + 비정상 종료 |

**원칙**: 에러가 나도 다른 이슈 처리에 영향을 주지 않는다. 이슈별 try-catch 격리.
