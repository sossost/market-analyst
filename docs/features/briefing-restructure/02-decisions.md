## DR-1: 데일리 브리핑 "시장 전망" 섹션 제거 여부

**Status**: accepted (현행 유지 — 이미 적용됨)

**Context**: 이슈 #108에서 데일리 리포트의 "시장 전망" 제거를 요청.

**분석**: `buildDailySystemPrompt()`를 검토한 결과, 현행 데일리 브리핑에는 이미 "시장 전망" 섹션이 없다. 데일리는 이미 지수 등락 + 시장 온도 + 특이종목 카탈리스트 중심의 데이터 브리핑 구조다. "핵심 데이터 변화 3가지"로 대체하는 명시적 변경도 불필요 — 현행 구조가 이미 그 역할을 한다.

**결정**: 데일리 시스템 프롬프트 구조 변경 없음. 이슈 #108의 데일리 관련 요청은 현행으로 충족된 것으로 판단.

## DR-2: 토론 브리핑 병목 섹션을 어디에 배치할 것인가

**Status**: accepted

**Options:**

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A (선택) | 섹션 3 "핵심 발견" 내에 병목 포함 | 병목이 구조적 해석의 일부로 자연스럽게 묶임 | 섹션이 길어질 수 있음 |
| B | 섹션 4 "기회" 안의 서브섹션으로 유지 | 현행과 변경 최소화 | 기회 섹션에 경고성 정보(해소 임박) 혼재 |
| C | 별도 섹션(8번째)으로 분리 | 가장 명확한 분리 | 섹션이 너무 많아짐 |

**결정**: 옵션 A 채택. 병목 상태는 구조적 해석의 핵심 근거이며, "왜 이 섹터가 기회인가"의 배경 설명으로 섹션 3에 자연스럽게 통합된다. 섹션 4는 순수하게 기회(진입 근거)에 집중하고, 병목의 해소 임박 신호는 섹션 3에서 먼저 짚은 후 섹션 5(경고)에서 구체화한다.

## DR-3: 서사 체인-종목 매칭 방식 선택

**Status**: accepted

**Options:**

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A (선택) | 프롬프트 주입 방식 — LLM이 체인과 종목 연결 판단 | 즉시 구현 가능, `beneficiarySectors` 데이터 불필요 | LLM 판단 오류 가능성 |
| B | 코드 레벨 매칭 — `beneficiarySectors`/`beneficiaryTickers` 비교 | 결정론적, 일관성 보장 | 현재 필드가 비어있어 구현 불가 |
| C | `parseBottleneckFromThesis()` 개선 후 필드 채우기 + 코드 매칭 | 장기적으로 가장 정확 | 스코프 과대화, 별도 이슈 필요 |

**결정**: 옵션 A 채택. `beneficiarySectors`는 `narrativeChainService.ts`에서 항상 `[]`로 저장되고 있어(파싱 로직이 빈 배열 반환) 코드 매칭이 현재 불가능하다. LLM 방식은 데이터 없이 즉시 작동하며, 오류도 허용 가능한 수준(태그가 잘못 붙는 것은 누락보다 낫다).

옵션 C(beneficiarySectors 채우기)는 별도 이슈로 추적. 이번 스코프에서 제외.

## DR-4: 데일리 서사 체인 포맷 — 주간과 다른 포맷 사용 여부

**Status**: accepted

**Context**: 주간 에이전트는 `formatChainsSummaryForPrompt()`로 병목 테이블(경과일, 참고 해소 기간 포함)을 받는다. 데일리에도 동일 포맷을 쓸지, 더 간결한 포맷을 만들지.

**결정**: 별도 함수 `formatChainsForDailyPrompt()` 추가. 이유:
1. 데일리의 목적은 "종목에 서사 태그 붙이기" — 주간의 "병목 모니터링 테이블"과 다르다.
2. 데일리 시스템 프롬프트는 이미 길다. 주간 포맷 그대로 주입하면 불필요한 컨텍스트(참고 해소 기간, 통계 안내)가 추가된다.
3. 함수 분리 → 각자 목적에 맞게 독립 진화 가능.

데일리 포맷: 체인명 + 메가트렌드 + 상태 + 경과일 (4컬럼). 간결하게.
