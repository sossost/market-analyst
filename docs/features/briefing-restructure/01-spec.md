**이슈**: #108 (투자 브리핑 구성 개편 7섹션) + #111 (데일리 리포트 서사 체인 태그 연결)
**날짜**: 2026-03-09
**작성**: mission-planner

## 선행 맥락

**현행 토론 브리핑 구조 (`round3-synthesis.ts` 분석):**

현재 moderator의 합성 프롬프트는 6개 섹션으로 구성된다:
1. 핵심 요약 (3줄 이내)
2. 시장 환경 판단 (숫자 + 의미 해석)
3. 핵심 발견 (1~2가지)
4. 주도섹터/주도주 전망 (부상하는 섹터, 병목 생애주기, 주의 섹터/종목 — 3개 서브섹션이 혼재)
5. 분석가 간 이견
6. 향후 주목할 이벤트

**현행 구조의 문제점 (이슈 #108 기반 분석):**

1. **섹션 4에 역할이 다른 3개 서브섹션이 혼재**: "부상하는 섹터"(기회)와 "주의해야 할 섹터/종목"(경고)과 "병목 생애주기"(서사 추적)가 같은 섹션에 뭉쳐있어 독자가 메시지를 즉시 파악하지 못한다.
2. **핵심 요약이 4개 주제를 섞음**: 3줄 요약에 구조적 변화 + 섹터/테마 + 리스크가 함께 묶여 "오늘 가장 중요한 것 1개"가 희석된다.
3. **섹션 2(시장 환경)에 팩트+해석이 혼재**: 수치와 해석이 섞여 있어 검증이 어렵다.
4. **이벤트 캘린더에 시간 순 정렬 규칙이 없음**: "다음 1~2주" 이벤트 나열이 시간 순으로 정렬되지 않아 일정 파악이 불편하다.

**현행 데일리 브리핑 구조 (`systemPrompt.ts` `buildDailySystemPrompt` 분석):**

- "시장 전망" 섹션이 없음 — 이미 데이터 중심 구조. 이슈 #108에서 요청한 "시장 전망 제거"는 현행에서 이미 적용된 상태.
- 특이종목 + Phase 1 후기/RS 상승 초기 예비군을 종목별로 나열.
- **서사 체인 연결 없음**: 이슈 #111 요청인 "종목에 체인 이름 + 상태 태그"가 현재 구현되지 않음.

**서사 체인 현황 (`narrativeChainService.ts`, `narrativeChainStats.ts`):**

- `narrative_chains` 테이블: megatrend, bottleneck, status(ACTIVE/RESOLVING/RESOLVED/OVERSUPPLY), beneficiarySectors(jsonb), beneficiaryTickers(jsonb) 컬럼 존재.
- 현재 `beneficiarySectors`와 `beneficiaryTickers`가 빈 배열(`[]`)로 저장됨 — 체인 생성 시 파싱하지 않아 채워지지 않는 상태.
- 주간 에이전트에는 `formatChainsSummaryForPrompt()`로 병목 테이블이 주입되지만, 종목 단위 매칭 로직은 없음.
- 데일리 에이전트에는 서사 체인 컨텍스트가 전혀 주입되지 않음.

**기존 관련 이슈 결과물:**
- Wave 2b (PR #101): `narrative_chains` 테이블 + `narrativeChainService.ts` + `narrativeChainStats.ts` 구현 완료.
- bull-bias 가드레일 (PR #114): 토론 브리핑과 데일리 브리핑 모두 Bull-Bias 섹션 추가됨.

## 골 정렬

**ALIGNED**

- **토론 브리핑 개편**: "오늘 가장 중요한 것 1개"를 즉시 파악하게 하면, 수신자가 주도섹터/주도주 포착 판단을 더 빠르게 내릴 수 있다. 정보 과부하를 줄이면 핵심 신호가 희석되지 않는다.
- **경고 섹션 분리**: 과열/위험 종목을 기회 섹션과 분리하면 잘못된 포지션 진입 리스크가 명시적으로 부각된다. 알파 형성에는 틀린 진입을 피하는 것도 포함된다.
- **서사 체인 태그**: 종목이 "어떤 구조적 서사에 올라타 있는지"를 즉시 알게 하면, 단순 기술적 지표를 넘어 서사 기반 판단이 가능해진다. Phase 2 상승 초입 종목이 어떤 병목 해소의 수혜를 받는지 명시하는 것은 골과 직접 연결된다.

## 문제

**토론 브리핑(round3-synthesis.ts):** 현재 6개 섹션 구조에서 기회·경고·병목이 뒤섞여 독자가 핵심 메시지를 즉시 파악하지 못하고, 이벤트 캘린더에 시간 순 정렬이 없다.

**데일리 브리핑(buildDailySystemPrompt):** 종목이 어떤 구조적 서사(narrative chain)와 연결되는지 태그가 없어, 기술적 데이터만 나열된 채 서사 맥락이 누락된다.

## Before → After

### 토론 브리핑

**Before (6섹션)**
1. 핵심 요약 (3줄 — 구조적 변화 + 섹터 + 리스크 혼재)
2. 시장 환경 판단 (수치 + 해석 혼재)
3. 핵심 발견 (1~2가지)
4. 주도섹터/주도주 전망 (기회 + 병목 + 경고 혼재)
5. 분석가 간 이견
6. 향후 주목할 이벤트 (시간 순 미보장)

**After (7섹션)**
1. 핵심 한 줄 (오늘 가장 중요한 것 단 1개)
2. 시장 데이터 (수치만, 해석 없이 — 팩트와 해석 분리)
3. 핵심 발견 + 병목 상태 (구조적 해석 — 기회의 근거)
4. 기회: 주도섹터/주도주 (테이블 + 서사)
5. 경고: 과열/위험 종목 (별도 분리)
6. 분석가 이견 (유지)
7. 이벤트 캘린더 (시간 순 정렬 명시)

### 데일리 브리핑

**Before:** 종목 나열 시 기술적 데이터만 (RS, Phase, MA150 기울기 등)

**After:** ACTIVE/RESOLVING 서사 체인과 매칭되는 종목에 `[체인명 / 상태]` 태그 추가
- 예: `NVDA RS 89 | Phase 2 | [AI인프라-GPU / ACTIVE]`

## 변경 사항

### Phase 1: 토론 브리핑 개편 (`round3-synthesis.ts`)

**파일**: `src/agent/debate/round3-synthesis.ts`
**함수**: `buildSynthesisPrompt()` 내 브리핑 구조 지시 섹션

**변경 내용:**

현행 `## 브리핑 구조 (아래 순서대로 작성)` 섹션을 7섹션으로 재작성.

```
### 1. 핵심 한 줄
오늘 가장 중요한 것 단 하나. 문장 1개, 50자 이내.
"~가 ~로 전환됨" 또는 "~가 ~를 촉발할 가능성 증가" 형식.

### 2. 시장 데이터 (수치만, 해석 없이)
- 주요 지수 종가 + 등락률
- 핵심 매크로 지표 수치 (VIX, 금리, 달러 등)
- 단순 수치 나열. 해석은 섹션 3에서.

### 3. 핵심 발견 + 병목 상태
- 분석을 통해 도출된 구조적 인사이트 1~2가지 + 각 근거
- 현재 추적 중인 병목의 상태 (ACTIVE/RESOLVING/RESOLVED)
  - RESOLVING 이상이면 "이탈 준비 시점 검토" 명시
  - N+1 병목 예측 (2명 이상 합의 시)

### 4. 기회: 주도섹터/주도주
우리의 목표는 상승 초입에 진입 중인 섹터와 종목을 남들보다 먼저 포착하는 것.

| 섹터/종목 | 근거 | 상태 |
|----------|------|------|
- 왜 지금인지 (촉매, 자금 흐름, 펀더멘탈)
- 관련 ETF 티커 + 대표 종목
- 5일/20일 가격 변화율 마이너스이면 "고점 피로감 경계" 표기
※ 목표가/손절가 없음. 구조적 변화의 초기 신호 포착이 목표.

### 5. 경고: 과열/위험 종목
- 모멘텀이 꺾이거나 과열 신호
- RS는 높지만 가격이 하락 중인 종목 (고점 피로감)
- 구조적 역풍을 맞는 산업
- 왜 지금 비중을 줄여야 하는지 근거

### 6. 분석가 이견 (있을 경우만)
- 의견이 갈리는 핵심 포인트 1가지
- 각 입장 2~3줄

### 7. 이벤트 캘린더
- 다음 1~2주 내 주요 이벤트를 날짜 순으로 정렬
- 형식: YYYY-MM-DD | 이벤트명 | 위 분석에 미치는 영향 (1줄)
```

**추가 지시:**
- 섹션 2는 수치만. 해석이 들어가면 섹션 3으로 이동.
- 섹션 4와 5는 반드시 별도 섹션. 경고 종목을 기회 섹션에 넣지 말 것.
- 섹션 7 이벤트는 날짜 오름차순 정렬 필수.
- 기존 "수치 정확성", "품질 기준", "용어 규칙" 지시는 유지.

## 리스크

### R1: `extractDebateOutput()` 정규식 충돌 (낮음)
새 섹션 헤더(`### 1. 핵심 한 줄` 등)가 `cleanReport` 생성 시 `replace` 패턴과 충돌할 가능성.
**대응**: 새 섹션명이 "검증 가능한 전망 추출" / "Thesis 추출" / "전망 추출" / "시장 레짐 판정" 패턴을 포함하지 않으므로 실제 충돌 위험은 없음. 단위 테스트로 검증.

### R2: `beneficiarySectors` 필드 비어있음 (기지 위험, 중간)
서사 체인의 `beneficiarySectors`와 `beneficiaryTickers`가 현재 비어있어 코드 레벨 매칭 불가.
**대응**: 이번 구현은 코드 매칭 방식이 아닌 LLM 판단 방식 채택. 프롬프트에 체인 정보를 주입하고 LLM이 종목-체인 연결을 판단. 향후 `beneficiarySectors` 필드가 채워지면 코드 레벨 매칭으로 업그레이드 가능.

### R3: 데일리 에이전트 컨텍스트 증가 (낮음)
서사 체인 주입으로 데일리 시스템 프롬프트 토큰 증가. 현재 데일리는 MAX_TOKENS 8192, MAX_ITERATIONS 15 설정.
**대응**: 체인이 없으면 주입 스킵. 체인 테이블은 간결한 포맷 (헤더 + 행 단위). 대부분의 경우 100~300 토큰 이내 증가 예상.

### R4: 토론 브리핑 품질 변화 (중간)
7섹션 재구조화 시 LLM이 기존 패턴 대신 새 구조를 따르는 데 적응 기간 필요.
**대응**: 기존 "수치 정확성", "품질 기준", "용어 규칙" 지시를 그대로 유지. 새 구조만 추가. 첫 실행 후 브리핑 품질 육안 검토 필요.

## 의사결정 필요

없음 — 바로 구현 가능.

**단, 구현 중 판단이 필요한 경우:**
- `beneficiarySectors` 필드를 이번에 채우는 로직(thesis 저장 시 파싱 개선)을 함께 구현할지 여부는 Phase 2 구현팀이 코드 검토 후 결정. 현재 `parseBottleneckFromThesis()`가 `beneficiarySectors: []` 반환 중. 개선 범위가 명확하면 함께 처리, 불명확하면 이번 스코프에서 제외하고 별도 이슈로.
