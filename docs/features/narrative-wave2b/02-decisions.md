## 결정 1: `narrative_chains` 테이블을 지금 만든다

### 컨텍스트

Wave 2a 스펙에서 이 결정을 명시적으로 연기했다:
> "프롬프트 변경으로 4주 이상 데이터를 축적한 뒤, nextBottleneck 패턴이 실제로 나타나는지 확인 후 Wave 2b에서 narrative_chains 도입을 검토하는 것이 올바른 순서다."

Wave 2a가 2026-03-08 머지됐고, Wave 2b는 즉시 착수한다.

### 선택지

**A. Wave 2a 데이터 4주 축적 후 결정**
- 장점: 테이블이 실제로 필요한지 확인 후 만들 수 있다.
- 단점: 4주 동안 병목 식별 날짜가 기록되지 않는다. 나중에 소급 기록할 수 없다. 데이터 손실.

**B. 즉시 테이블 생성 + 데이터 축적 시작**
- 장점: `bottleneck_identified_at`의 정확도가 높아진다. 빠를수록 유리한 시계열 데이터.
- 단점: 당장 데이터가 없어서 통계적 의미가 없다.

### 결정

**B 채택.** 시계열 데이터는 시작이 늦을수록 손해다. `bottleneck_identified_at`을 나중에 추정해서 넣는 것은 신뢰할 수 없다. 테이블을 만들어 놓고 축적하면 된다. 단, 체인 3개 미만 동안은 통계를 노출하지 않는다(null 반환)는 가드를 구현하여 의미 없는 데이터가 리포트에 주입되는 것을 방지한다.

## 결정 2: Wave 2b 범위를 핵심 필드로 한정한다

### 컨텍스트

RFC 1-C는 `cross_trends`, `policy_signal_ids` 등 풍부한 필드를 제안했다. 이 모두를 Wave 2b에서 구현할 수 있지만, 그럴 필요가 있는가?

### 선택지

**A. RFC 1-C 전체 구현 (모든 필드 포함)**
- 장점: 한 번에 끝난다.
- 단점: `cross_trends`는 체인 5개 이상이 있어야 의미 있다. 지금 구현하면 빈 필드를 넣는 꼴이다. `policy_signal_ids`는 `policy_signals` 테이블이 없으면 외래키를 연결할 대상이 없다.

**B. 핵심 필드만 구현 (날짜 추적 + 병목 상태 + 수혜 기업 + thesis 연결)**
- 장점: Wave 2b의 핵심 가치("날짜 기록")에 집중. 불필요한 복잡도 없음.
- 단점: 나중에 마이그레이션을 추가해야 할 수 있다.

### 결정

**B 채택.** 지금 당장 필요한 것은 날짜 데이터다. 빈 필드를 미리 만드는 것은 스키마 복잡도만 늘린다. `cross_trends`와 `policy_signal_ids`는 RFC 상에서는 Wave 4에 해당하는 내용으로, 기반 데이터가 쌓인 후에 추가하는 것이 맞다.

## 결정 3: 중복 체인 감지를 유사 매칭으로 처리한다

### 컨텍스트

LLM은 동일한 병목 개념을 다양한 표현으로 작성한다:
- "광트랜시버 공급 부족"
- "Optical Transceiver 병목"
- "광모듈 공급망 제약"

이들이 같은 체인인지 다른 체인인지를 시스템이 판단해야 한다.

### 선택지

**A. 완전 문자열 일치**
- 장점: 구현 단순.
- 단점: 표현이 조금만 달라도 다른 체인으로 인식. 체인이 폭발적으로 증가.

**B. 유사 매칭 (임계값 기반)**
- 장점: 의미적으로 같은 체인을 하나로 묶을 수 있다.
- 단점: 임계값 설정이 주관적. 오탐/미탐 가능.

**C. LLM 기반 의미 동일성 판단**
- 장점: 가장 정확.
- 단점: 체인 기록마다 LLM 호출. 비용과 지연 발생.

### 결정

**B 채택.** 초기에는 `megatrend` 동일 + `bottleneck` 텍스트 70% 이상 겹침(Jaccard 유사도 또는 단어 교집합 기반)으로 구현한다. 4주 운영 후 중복 체인 발생 패턴을 점검하여 임계값을 조정하거나, 필요 시 C로 전환한다. A는 채택하지 않는다 — 완전 일치는 데이터 품질을 심각하게 떨어뜨린다.

## 결정 4: 체인 기록 실패를 에러 격리한다

### 컨텍스트

`thesisStore.ts`의 thesis 저장 흐름에 체인 기록을 추가할 때, 체인 기록이 실패하면 어떻게 처리할 것인가?

### 선택지

**A. 체인 기록 실패 시 thesis 저장 전체를 롤백**
- 장점: 데이터 일관성.
- 단점: 체인 기록 버그로 인해 토론 결과(thesis)가 저장되지 않는 사태 발생 가능. 핵심 기능이 부가 기능에 종속.

**B. 체인 기록 실패 시 로그만 남기고 thesis 저장은 계속 진행**
- 장점: 핵심 기능(thesis 저장)이 부가 기능(체인 기록)의 실패에 영향받지 않음.
- 단점: 체인 데이터 누락 가능성. 단, 누락은 다음 토론에서 보완 가능.

### 결정

**B 채택.** thesis는 토론의 핵심 결과물이고, 체인은 이를 기반으로 한 부가 분석이다. 부가 기능의 실패가 핵심 기능을 멈추게 해서는 안 된다. 체인 기록 실패는 `logger.warn`으로 기록하되 예외를 삼킨다.

## 결정 5: 통계 최소 샘플 수를 3개로 설정한다

### 컨텍스트

체인이 몇 개 해소되었을 때 "평균 해소 기간"을 산출하여 리포트에 노출할 것인가?

### 선택지

**A. 1개부터 노출** — 단일 데이터포인트도 참고값
**B. 3개부터 노출** — 최소한의 분포를 볼 수 있는 기준
**C. 5개부터 노출** — 통계적 유의성 확보 관점

### 결정

**B 채택.** RFC 이슈 본문("체인 3개+ 이후 의미 있음")과 일치. 1개는 패턴이 아니라 사례다. 3개면 최솟값/최댓값/평균이 의미를 가지기 시작한다. 5개는 너무 보수적 — 체인이 주 1~2회 생성된다고 해도 5개 해소까지 수개월이 걸릴 수 있다. 3개를 기준으로 하되, 표준편차가 크면 "데이터 이질성 주의" 문구를 추가한다.
