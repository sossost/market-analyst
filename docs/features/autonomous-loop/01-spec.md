# Spec: 자율 개선 루프 (Autonomous Improvement Loop)

## 철학

모든 에이전트는 동일한 구조를 따른다:

```
관찰 → 판단 → 실행 → 결과 측정 → 기록 → 기준 개선 → 관찰
```

사람은 **방향만 설정**하고, 시스템이 이 루프를 자율적으로 돈다.
사람에게 올라가는 건 **결과 리포트**뿐이다.

## 목적

"Phase 2 상승 초입의 주도섹터/주도주를 남들보다 먼저 포착"하는 시스템이
사람 개입 없이 스스로 품질을 개선해나가는 것.

## 핵심 원칙

1. **모든 것은 같은 루프다** — 시그널 튜닝, 토론 품질, 리포트 품질, 매니저 판단 전부 동일 구조
2. **측정 없이 개선 없다** — 모든 판단에는 측정 가능한 성과 지표가 있어야 한다
3. **자동으로 돌아야 한다** — 크론으로 실행, 결과 기록, 다음 실행에 반영
4. **사람은 방향만** — CEO에게 올라가는 건 "이번 주 성과 요약 + 시스템이 스스로 바꾼 것 + 의사결정 필요한 것"

## 자율 루프 구조

### Layer 1: 시그널 루프 (기계적)

```
매일 크론:
  1. Phase 2 진입 시그널 감지
  2. 기존 시그널의 수익률 업데이트 (이미 구현됨: update-recommendation-status)
  3. 주간: 파라미터별 성과 집계 (backtest-signals 로직)
  4. 성과 기반 파라미터 자동 조정
     - RS 임계값: 성과 좋은 구간으로 수렴
     - 거래량 필터: on/off 성과 비교 후 자동 결정
     - 섹터 필터: 섹터별 성과 기반 가중치
  5. 조정 내역 기록 (언제, 왜, 뭘 바꿨는지)
```

**측정 지표**: 20일 수익률, 승률, Phase 종료 수익률
**비용**: $0 (순수 DB 연산)

### Layer 2: 분석 루프 (LLM)

```
매일 크론:
  1. 애널리스트 토론 실행 (이미 구현됨)
  2. Thesis 생성 → 검증 → 원인 분석 (이미 구현됨)
  3. 주간: 애널리스트별 적중률 집계
     - 누가 맞았고 누가 틀렸는지
     - 어떤 유형의 전망이 잘 맞는지
  4. 프롬프트 자동 튜닝
     - 적중률 높은 애널리스트의 분석 패턴을 강화
     - 반복 실패 패턴을 프롬프트에 경고로 주입
  5. 튜닝 내역 기록
```

**측정 지표**: Thesis 적중률, 애널리스트별 성과
**비용**: ~$21/월 (기존 운영비)

### Layer 3: 전달 루프 (리포트)

```
매일:
  1. Layer 1 시그널 + Layer 2 분석 → 리포트 생성
  2. Discord 발송
주간:
  3. CEO 리포트 생성
     - 이번 주 시그널 성과
     - 시스템이 자동 조정한 내역
     - 의사결정 필요 사항 (있으면)
```

### Layer 4: 메타 루프 (매니저)

```
주간:
  1. 전체 시스템 상태 점검
     - Layer 1-3이 정상 동작하는가
     - 성과가 개선되고 있는가, 악화되고 있는가
  2. 의사결정 기록 리뷰
     - 이번 주 자동 조정들이 합리적이었는가
  3. CEO 리포트에 "매니저 판단" 섹션 포함
     - "시그널 품질이 개선/악화 추세입니다"
     - "X 파라미터 조정이 필요할 수 있습니다"
  4. chief-of-staff.md 업데이트
```

## 데이터 흐름

```
[크론: 매일]
  ETL → stock_phases, sector_rs
  시그널 감지 → signal_log 테이블 (신규)
  기존 시그널 업데이트 → signal_log.returns 업데이트
  토론 → theses → 검증

[크론: 매주]
  시그널 성과 집계 → signal_performance 테이블 (신규)
  파라미터 자동 조정 → signal_params 테이블 (신규)
  애널리스트 성과 집계 → agent_performance (신규)
  CEO 리포트 생성 → Discord #ceo-brief 채널

[크론: 매월]
  전체 성과 트렌드 분석
  파라미터 대폭 조정 여부 판단
```

## 신규 테이블

### signal_log
시그널 발생 → 추적 → 결과까지 전 생애주기.

| 컬럼 | 설명 |
|------|------|
| symbol, entry_date | 시그널 식별 |
| entry_price | 진입가 |
| params_snapshot | 당시 적용된 파라미터 (JSON) |
| rs_score, volume_confirmed, sector_group_phase | 진입 시점 팩터 |
| return_5d, return_10d, return_20d, return_60d | 고정 기간 수익률 |
| phase_exit_date, phase_exit_return | Phase 종료 시점 |
| max_return | 보유 중 최대 수익 |
| status | ACTIVE / CLOSED |

### signal_params
현재 적용 중인 시그널 파라미터 + 변경 이력.

| 컬럼 | 설명 |
|------|------|
| param_name | rs_threshold, volume_required, ... |
| current_value | 현재 값 |
| previous_value | 이전 값 |
| change_reason | 왜 바꿨는지 |
| changed_at | 변경 시점 |
| performance_before / after | 변경 전후 성과 |

## Scope

### In scope
- Layer 1 (시그널 자동 튜닝) 전체
- Layer 2 (애널리스트 성과 집계) — 프롬프트 자동 튜닝은 Phase 2
- Layer 3 (CEO 주간 리포트)
- Layer 4 (매니저 메타 판단) — 기록 구조

### Out of scope
- LLM 프롬프트 완전 자동 튜닝 (사람 검토 필요)
- 자동 매매 연동
- 외부 사용자 대시보드

## 성공 기준

1. 4주간 사람 개입 없이 시그널 파라미터가 자동 조정됨
2. CEO 주간 리포트에 "이번 주 시스템이 뭘 바꿨는지"가 명확히 나옴
3. 시그널 승률이 4주 후 초기 대비 측정 가능하게 변화함 (개선이든 악화든 — 측정 자체가 성공)
