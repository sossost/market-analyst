# Ideation: 에이전트 토론 & 진화 시스템 (내각 시스템)

> 각 분야 전문 에이전트("장관")를 영구 구축 → 토론 → 인사이트 생성 → 시장 검증 → 장기 기억 축적
> 시간이 지날수록 에이전트의 분석 수준이 우상향하는 구조

---

## 문제 정의

현재 에이전트는 매번 같은 수준에서 시작한다.
- 컨텍스트 윈도우 안에서만 사고 → 세션 끝나면 리셋
- 과거에 했던 예측이 맞았는지/틀렸는지 모름
- 단일 관점 분석 → 편향에 취약

**원하는 상태**: 에이전트가 토론과 시장 검증을 통해 경험을 축적하고, 분석 프레임워크 자체가 진화하는 시스템.

---

## 핵심 컨셉

### 0. 장관 시스템 (`.claude/agents/` 기반 영구 페르소나)

각 분야 전문가를 Claude Code 커스텀 에이전트로 **영구 구축**한다.
토론 세션뿐 아니라 평소에도 언제든 불러서 자문할 수 있는 "내각".

**구현 방식**: `.claude/agents/{name}.md` 마크다운 파일 (YAML frontmatter + system prompt)

```yaml
# .claude/agents/macro-economist.md
---
name: macro-economist
description: 매크로 이코노미스트. 금리/통화정책/경기사이클/유동성 관점에서 시장을 분석. 거시경제 질문이나 토론 시 사용.
model: sonnet
memory: project          # 세션 간 기억 축적
tools: Read, Grep, Glob, Bash, WebSearch, WebFetch
---

당신은 매크로 이코노미스트입니다.
...전문 system prompt...
```

**핵심 설계 원칙**:

| 원칙 | 설명 |
|------|------|
| **영속성** | `.claude/agents/`에 커밋 → 프로젝트와 함께 버전 관리 |
| **`memory: project`** | 세션 간 학습 축적 — 과거 토론, 검증 결과, 자신의 적중/빗나감 이력을 기억 |
| **독립 소환 가능** | 토론 없이도 "매크로 이코노미스트에게 물어봐" 식으로 단독 자문 |
| **도구 접근 차별화** | 에이전트별로 필요한 도구만 부여 (최소 권한) |
| **모델 선택** | 일상 자문은 Sonnet, 심층 토론은 Opus로 유연하게 |

**장관 후보 & 역할**:

```
.claude/agents/
├── macro-economist.md     # 매크로 이코노미스트 — 탑다운 거시경제 (구축 완료)
├── tech-analyst.md        # 테크/산업 애널리스트 — 바텀업 기술 트렌드 (구축 완료)
├── geopolitics.md         # 지정학 전략가 — 구조적 변화, 공급망 (구축 완료)
├── sentiment-analyst.md   # 시장 심리 분석가 — 자금 흐름, 행동경제학 (구축 완료)
└── moderator.md           # 모더레이터 — 토론 종합, 합의/불일치 정리 (구축 완료)
```

**2레이어 기억 구조**:

```
Layer 1: memory: project (Claude Code 내장)
  - 세션 간 자동 축적
  - 에이전트별 개인 기억 (.claude/memory/agents/{name}/)
  - "지난번에 내가 X를 예측했는데..." 식의 자기 참조 가능

Layer 2: DB 기반 Thesis Ledger (아래 섹션 2)
  - 구조화된 예측 데이터
  - ETL 데이터와 자동 대조
  - 적중률 통계, 패턴 추출
```

> **열린 질문**: `memory: project`의 실제 저장 위치와 크기 제한 — 토큰 예산이 얼마인지 확인 필요.
> **열린 질문**: 에이전트 간 기억 공유 — 각자의 기억은 독립적인데, 토론에서 나온 합의는 어떻게 공유할지.
> **열린 질문**: 장관을 추가/교체하는 기준 — 예를 들어 "에너지 전문가"가 필요해지면?

### 1. 멀티 페르소나 토론

서로 다른 전문성을 가진 에이전트가 같은 주제를 놓고 토론.
단일 프롬프트 대비 블라인드스팟 감소, 다각도 분석.

**페르소나 후보**:

| 페르소나 | 관점 | 주요 분석 축 |
|----------|------|-------------|
| 매크로 이코노미스트 | 탑다운 | 금리, 통화정책, 경기사이클, 유동성 |
| 테크/산업 애널리스트 | 바텀업 | 기술 트렌드, capex 사이클, 혁신 채택 곡선 |
| 지정학 전략가 | 구조적 변화 | 무역, 공급망, 규제, 정치 리스크 |
| 시장 심리 분석가 | 행동경제학 | 자금 흐름, 포지셔닝, 센티먼트, 군중 심리 |

> **열린 질문**: 4명이 적당한가? 3명이면 부족하고 5명이면 비용이 늘어남.
> **열린 질문**: 모더레이터를 별도 에이전트로 둘지, 마지막 라운드에서 한 명이 겸임할지.

### 2. Thesis Ledger (예측 원장)

토론에서 나온 주장을 **검증 가능한 형태**로 구조화해서 DB에 기록.

```
예시:
- thesis: "2분기 금리 인하 시 중형 품질주가 대형 성장주보다 먼저 반응"
- agent: macro_economist
- timeframe: 90일
- verification_metric: "mid_cap_quality RS vs large_cap_growth RS"
- target_condition: "mid_cap_quality RS > large_cap_growth RS by 5%+"
- status: ACTIVE → CONFIRMED / INVALIDATED / EXPIRED
```

핵심: **모호한 전망이 아니라 시장 데이터로 검증 가능한 명제**여야 함.

> **열린 질문**: thesis 구조화를 에이전트가 직접 하게 할지, 별도 "구조화 에이전트"가 할지.
> **열린 질문**: timeframe을 에이전트가 자유롭게 정하게 할지, 30/60/90일 중 선택하게 할지.

### 3. 자동 검증 루프

기존 ETL 데이터와 과거 thesis를 정기적으로 대조:

```
매주 실행:
1. ACTIVE thesis 중 timeframe 도래한 건 조회
2. 해당 metric의 실제 시장 데이터 조회
3. target_condition 충족 여부 판정
4. CONFIRMED / INVALIDATED 업데이트 + 원인 분석 기록
```

이미 있는 것과의 연결:
- `recommendation_factors` (Issue #9A) — 종목 추천의 예측→검증 패턴
- 섹터/산업 RS 데이터 — thesis 검증에 바로 활용 가능
- `searchCatalyst` — 검증 시점에 관련 뉴스 검색으로 원인 분석 보강

> **열린 질문**: 검증 판정을 규칙 기반으로 할지, 에이전트가 판단하게 할지.
> **열린 질문**: "부분 적중"을 어떻게 처리할지 (방향은 맞았지만 크기가 다른 경우 등).

### 4. Evolving Memory (진화하는 장기 기억)

검증 결과를 바탕으로 **원칙/패턴**을 추출해서 장기 기억에 축적.

```
승격 조건: 동일 패턴이 3회 이상 적중

[검증된 원칙]
- "금리 인하 사이클 초기 → 중형 품질주 선행" (적중 4/5회)
- "섹터 RS 상위 진입 후 3주 내 거래량 미확인 → fake rotation 70%" (적중 3/4회)

[경계 패턴]
- "VIX 30+에서 바닥론 → 2/3이 조기 판단" (빗나감 4/6회)
- "지정학 이벤트 후 공급망 수혜주 추천 → 1주 반응 후 되돌림 빈번" (빗나감 3/5회)
```

이 장기 기억이 **다음 토론의 system prompt에 주입**됨 → 같은 실수 반복 방지.

> **열린 질문**: 장기 기억의 최대 크기를 제한할지 (토큰 예산).
> **열린 질문**: 오래된 원칙의 decay — 시장 레짐이 바뀌면 과거 원칙이 오히려 해가 될 수 있음.


## 리스크 & 고려사항

1. **Echo Chamber**: 에이전트끼리 서로 동의만 하는 현상
   - 대응: 페르소나에 "반드시 다른 관점 제시" 지시 포함, Devil's Advocate 역할 명시

2. **기억 오염**: 잘못된 원칙이 장기 기억에 들어가면 이후 모든 토론을 오염
   - 대응: 승격 조건 엄격화 (3회 이상 적중), 정기 감사, decay 메커니즘

3. **Overfitting**: 과거 패턴에 과적합 → 레짐 변화 시 역효과
   - 대응: 장기 기억에 "유효 기간" 또는 "시장 레짐 태그" 부여

4. **검증의 모호성**: "부분 적중"이나 "방향은 맞지만 타이밍 틀림" 처리
   - 대응: 이진 판정 대신 점수제 (0~1) 고려

5. **비용 누적**: 에이전트 수 × 라운드 수에 비례
   - 대응: Sonnet 기본, 핵심 라운드만 Opus

## 시운전 결과 (2026-03-05)

### 테스트 조건
- 질문: "2026년 3월 현재, 향후 3개월간 미국 주식시장에서 돈이 몰릴 섹터는 어디이고 왜인가?"
- 4명 병렬 실행 (general-purpose 에이전트 + 장관 프롬프트 주입)
- 모델: Sonnet, 웹 검색 활용

### 결과 요약

| 장관 | 핵심 고유 인사이트 | 프롬프트 품질 |
|------|-------------------|-------------|
| 매크로 | 워시 Fed 의장 지명 → 베어 스티프닝, ISM/CPI 실제 수치 활용 | 우수 |
| 테크 | ASIC vs GPU 구조 전환, Capex $490B 밸류체인 분해 | 우수 (범위 과대 → 수정 완료) |
| 지정학 | IEEPA 위헌 → Section 122 전환의 150일 시한 | 우수 |
| 심리 | Mag7 재귀성 루프 역전, VIX/AAII/ETF flows 정량화 | 우수 (틀릴 조건 약함 → 수정 완료) |

### 확인된 강점
- 4명 모두 관점 차별화 뚜렷 — 같은 "방산 강세"도 접근 각도가 다름
- 각자만 발견한 고유 인사이트 존재 → 토론 시 교차 검증 가치 높음
- 웹 검색 + 프레임워크 결합이 잘 작동

### 발견된 개선점 (프롬프트 수정 반영 완료)
1. 테크: 헬스케어까지 다뤄서 범위 초과 → "분석 범위" 섹션 추가
2. 테크/심리: "틀릴 조건 명시" 약함 → 사고 원칙에 추가
3. 합의가 너무 쉽게 형성 (4/4 방산 동의) → 모더레이터 역할로 해결 예정

### 열린 질문 해소
- [x] 장관별 system prompt 깊이 → 상세한 분석 프레임워크가 효과적 (확인)
- [x] 초기 장관 구성 → 4명 + 모더레이터가 적절 (차별화 충분, 비용 합리적)
