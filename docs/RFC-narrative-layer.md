# RFC: 시스템 진화 — 서사 레이어, 신호 다각화, 메타 학습

**Status:** Draft
**Date:** 2026-03-08
**Author:** jangyoonsu + Claude review session

---

## 배경

현재 Market Analyst 시스템은 기술적 신호(Phase, RS, 거래량) + 펀더멘탈(SEPA) 기반으로 Phase 2 초입 종목을 스크리닝한다. Layer 1~8 + Phase A까지 완료된 상태.

### 핵심 문제 인식

PLTR(23), NVDA(24), MU/LITE(25) 같은 알파 종목은 기술적 신호가 나타나기 **이전에** 구조적 서사(메가트렌드 → 수요 폭발 → 공급 체인 병목)가 먼저 존재했다. 이 수요-공급-병목 체인은 AI 고유의 현상이 아니라 90년대 인터넷(광케이블), 스마트폰(AP/디스플레이), 코로나 리오프닝(반도체 장비), 에너지 전환(리튬) 등 **모든 메가트렌드에서 반복된 범용 구조**다.

현재 시스템은 이 서사적 맥락을 체계적으로 다루지 않으며, 기술적/펀더멘탈 신호 외의 선행 신호 소스(정책, 경영진 행동, 섹터 간 시차)도 활용하지 않고 있다.

### 이 RFC의 범위

**Part 1 — 서사 레이어 + 검증 강화**
1. 서사 레이어 추가 (수요-공급-병목 프레임)
2. 병목 생애주기 추적 (이동 속도, 공급 과잉 전환, N+1 예측)
3. 위양성 비용 추적
4. 학습 유효성 검증 (Holdout Testing)

**Part 2 — 신호 다각화**
5. 정책-규제 선행 신호
6. 경영진 행동 패턴
7. 섹터 간 시차(Lag) 패턴
8. 실패 패턴 체계적 축적

**Part 3 — 메타 학습**
9. 크로스 메가트렌드 교차점
10. 시장 레짐 분류기
11. 애널리스트 합의도 추적
12. 시스템 메타 인사이트 수집

## 제안 1: 서사 레이어 — 수요-공급-병목 프레임

### 목표

토론 에이전트가 "시장 상황 논평"이 아니라 **"구조적 수요 변화 → 공급 체인 → 병목 → 수혜 기업"** 체인을 도출하도록 프레임을 전환한다.

### 기존 시스템과의 관계

대체가 아니라 **레이어 추가**. 기존 파이프라인(Phase + RS + 펀더멘탈 스크리닝)은 그대로 유지하고, 서사 레이어의 출력과 교집합을 취한다.

```
기존 파이프라인 → Phase 2 초입 후보 15개
서사 레이어   → 병목 체인 수혜 종목 10개
                         ↓
                   교집합 3~5개 = 최우선 관심 대상
```

"교집합이 최강 필터"라는 기존 발견의 연장선.

### 구현 방안

#### 1-A. 토론 프롬프트 프레임 추가

4명의 애널리스트 토론에 **수요-공급-병목 분석 프레임**을 필수 항목으로 추가.

```
현재 토론 프레임:
  라운드 1: 독립 분석 (매크로, 테크, 지정학, 심리)
  라운드 2: 교차 검증 + 반박
  라운드 3: 종합

추가할 프레임 (라운드 1에 삽입):
  각 애널리스트는 자신의 관점에서 다음을 반드시 다룰 것:
  1. 현재 가장 큰 구조적 수요 변화는 무엇인가?
  2. 그 수요를 충족시키는 공급 체인은 어떻게 구성되어 있는가?
  3. 병목은 어디인가? (현재 또는 예상)
  4. 그 병목을 해소하는 기업/섹터는?
```

#### 1-B. Thesis 카테고리 분리

현재 thesis 테이블에 `category` 필드 추가:

| category | 설명 | 기본 timeframe |
|----------|------|---------------|
| `structural_narrative` | 수요-공급-병목 서사 | 8~12주 |
| `sector_rotation` | 섹터 로테이션 전망 | 2~4주 |
| `short_term_outlook` | 단기 시장 전망 | 1~2주 |

구조적 서사 thesis는 검증 주기를 길게 잡아야 한다. 현재 thesis 만료가 timeframe 기반이므로, category별 기본 timeframe 설정만 추가하면 기존 인프라에서 작동.

#### 1-C. 서사 체인 DB 저장

새 테이블 또는 기존 테이블 확장:

```
narrative_chains:
  id, created_at,
  megatrend        -- "AI 인프라 확장"
  demand_driver    -- "데이터센터 GPU 수요 3배 증가"
  supply_chain     -- "GPU → HBM → 광트랜시버 → 전력"
  bottleneck       -- "광트랜시버 공급 부족"
  bottleneck_identified_at  -- 병목 최초 식별일
  bottleneck_resolved_at    -- 병목 해소 확인일 (nullable)
  next_bottleneck           -- 예측된 다음 병목 (nullable)
  beneficiary_sectors   -- ["Optical Components", "Fiber Optics"]
  beneficiary_tickers   -- ["LITE", "COHR", "II-VI"]
  status           -- ACTIVE / RESOLVING / RESOLVED / OVERSUPPLY / INVALIDATED
  linked_theses    -- [thesis_id, ...]
  cross_trends     -- 다른 megatrend와의 교차 (nullable)
```

이 체인이 기술적 스크리닝 결과와 교차될 때 신호 강도가 높아진다.

#### 1-E. 병목 생애주기 추적

병목은 생성 → 심화 → 해소 → 공급 과잉의 생애주기를 가진다. 90년대 광케이블의 5% 미만만 사용된 "다크 파이버", 리튬 가격 급등 후 공급 과잉 조정 등 모든 역사적 사례에서 동일 패턴.

```
병목 생애주기:
  ACTIVE     → 병목 존재, 수혜주 상승 구간
  RESOLVING  → 설비 투자 발표 급증, CAPEX 뉴스 다수 감지
  RESOLVED   → 공급 충족, 수혜주 모멘텀 둔화
  OVERSUPPLY → 공급 과잉, 수혜주 하락 위험

전환 신호 (토론 에이전트가 뉴스에서 감지):
  ACTIVE → RESOLVING: 경쟁사 진입 발표, 대규모 CAPEX 계획, 신규 공장 착공
  RESOLVING → RESOLVED: 리드타임 정상화, 가격 하락 시작
  RESOLVED → OVERSUPPLY: 재고 축적 뉴스, 가격 급락, 가동률 하락
```

축적 효과: `bottleneck_identified_at`과 `bottleneck_resolved_at`의 차이를 기록하면, "이 유형의 병목은 평균 N개월 후 해소된다"는 패턴이 축적된다. 이는 진입뿐 아니라 **이탈 타이밍** 판단에 사용.

#### 1-F. N+1 병목 예측

현재 병목이 해소되면 공급 체인의 다음 노드가 병목이 된다. GPU → 메모리 → 광통신 → 전력 → 냉각. 이 추론은 공급 체인 구조를 알면 논리적으로 도출 가능하며, LLM이 잘 수행할 수 있는 작업.

```
구현:
  - narrative_chains.supply_chain에서 현재 bottleneck 이후의 노드를 추출
  - 토론 라운드에 "현재 병목이 해소된다면 다음 제약은 어디인가?" 질문 추가
  - next_bottleneck 필드에 기록, 실제 발생 시 적중률 추적
```

#### 1-G. 크로스 메가트렌드 교차점

단일 트렌드보다 두 개의 메가트렌드가 교차하는 지점에서 더 큰 알파가 발생한다.

```
예시:
  AI + 에너지 전환 → 데이터센터 전력/냉각 → Vertiv, Eaton
  AI + 바이오 → 신약 개발 가속 → CRO/CDMO 기업
  AI + 국방 → 자율무기/감시 → Palantir, L3Harris

구현:
  - narrative_chains에 cross_trends 필드 추가
  - 서사 체인이 5개 이상 축적되면, 체인 간 공통 수혜자 자동 검색
  - "2개 이상의 서사 체인에서 수혜자로 등장" = 강한 교집합 신호
```

#### 1-D. 학습 단위 전환

현재: "이 종목 추천이 맞았나" → 사례 수가 적어 통계적 유의성 확보 어려움.

추가: **"이 서사가 실제 시장 반응으로 이어졌나"** → 학습 가능 사례 증가.

```
학습 예시:
  서사: "AI → 메모리 병목" (2024-Q2 생성)
  검증: MU +85% (6개월), SK하이닉스 +60% → CONFIRMED
  메타 패턴 승격: "수요 폭발 → 2차 공급자 병목" 패턴 유효
  → 다음 유사 구조 발생 시 프롬프트에 주입
```

### 난이도 및 일정 추정

| 항목 | 난이도 | 예상 일수 |
|------|--------|----------|
| 1-A. 프롬프트 프레임 추가 | 낮음 | 0.5일 |
| 1-B. Thesis 카테고리 분리 | 낮음 | 0.5일 |
| 1-C. 서사 체인 DB | 중간 | 1~2일 |
| 1-D. 학습 단위 확장 | 중간 | 1~2일 |
| 1-E. 병목 생애주기 추적 | 중간 | 1일 |
| 1-F. N+1 병목 예측 | 낮음 | 0.5일 (프롬프트 추가) |
| 1-G. 크로스 트렌드 교차점 | 낮음 | 0.5일 (체인 축적 후 자동화 1일) |

## 제안 2: 위양성 비용 추적

### 문제

현재 시스템은 적중률(CONFIRMED / total)을 추적하지만, **위양성의 비용**을 명시적으로 측정하지 않는다. Phase 2 전환율 41.9%는 58.1%가 실패한다는 의미이며, 이 실패 케이스가 리포트 수신자에게 어떻게 전달되는지가 실질적 신뢰도를 결정한다.

### 구현 방안

추천 성과 추적 시 다음 지표 추가:

```
false_positive_tracking:
  - phase2_signal_then_reverted: Phase 2 진입 신호 후 Phase 1로 회귀한 비율
  - revert_market_conditions: 회귀 시점의 시장 조건 (breadth, index trend 등)
  - max_adverse_move: 추천 후 최대 역행 폭
  - time_to_revert: 신호 발생 ~ 회귀까지 소요 기간
```

주간 리포트에 "이번 주 위양성 케이스" 섹션 추가. 맞은 것만 보여주는 것보다 틀린 것을 투명하게 보여주는 것이 시스템 신뢰도를 높인다.

### 난이도: 낮음~중간 | 예상 1~2일

기존 recommendations 테이블 + Phase 변화 추적 인프라 활용 가능.

## 제안 3: 학습 유효성 검증 (Holdout Testing)

### 문제

학습 루프가 예측력을 실제로 높이는지, 특정 시장 레짐에 과적합하는 건지 구분할 방법이 현재 없다. 6개월 만료 메커니즘이 있지만, 레짐 전환은 갑자기 오기 때문에 사전 감지가 필요.

### 구현 방안

월 1회 "홀드아웃 주간" 실행:

```
holdout_test:
  - 해당 주의 토론 1회를 학습 주입 없이 실행 (baseline)
  - 동일 데이터로 학습 주입 버전도 실행 (treatment)
  - 두 결과의 thesis 품질/적중률을 비교
  - 차이가 유의미하면 학습이 효과적
  - 차이가 없거나 역전되면 과적합 경고
```

비용: 토론 1회 추가 (~$0.5-1.0/월). 학습 루프 전체의 유효성을 검증하는 데 비하면 저렴.

### 난이도: 중간 | 예상 2~3일

에이전트 루프에 `--no-learnings` 플래그 추가 + 비교 리포트 생성.

## 제안 4: 정책-규제 선행 신호 추적

### 근거

시장을 만드는 건 기술만이 아니라 정책이다. 1996년 통신법 → 텔레콤 붐, IRA → 클린에너지 투자 폭발, CHIPS Act → 반도체 리쇼어링. **법안이 통과되는 시점이 아니라, 발의되고 위원회를 통과하는 시점**이 진짜 선행 신호.

### 구현 방안

```
토론 프롬프트에 정책 프레임 추가:
  - 현재 입법 파이프라인에서 시장에 구조적 영향을 줄 법안은?
  - 이 법안이 통과되면 어떤 섹터가 구조적 수혜를 받는가?
  - 관세/무역 정책 변화의 공급 체인 영향은?

policy_signals 테이블:
  id, created_at,
  policy_name       -- "CHIPS Act Phase 2 확장"
  status            -- PROPOSED / COMMITTEE / PASSED / ENACTED
  affected_sectors  -- ["Semiconductors", "Equipment"]
  affected_tickers  -- ["INTC", "AMAT", "LRCX"]
  linked_narratives -- [narrative_chain_id, ...]
  
뉴스 수집 시 정책/규제 카테고리 필터 추가:
  Brave Search 쿼리에 "bill", "regulation", "executive order", 
  "tariff", "subsidy" 등 정책 키워드 포함
```

### 난이도: 낮음~중간 | 예상 1~2일

토론 프롬프트 수정 + 뉴스 쿼리 확장. DB 저장은 서사 체인과 유사 구조.

## 제안 5: 경영진 행동 패턴 분석

### 근거

경영진이 **말하는 것**보다 **하는 것**이 더 정직하다. 두 가지 신호 소스:

1. **내부자 매매**: Phase 1 후기에서 내부자 매수가 나타나는 경우 = 매우 강한 신호
2. **컨퍼런스콜 톤 변화**: "cautiously optimistic" → "significant demand" 같은 언어 변화를 LLM이 잘 감지 가능

이것은 기술적 분석도 펀더멘탈도 아닌 **제3의 신호 소스**.

### 구현 방안

```
Phase 1: 컨퍼런스콜 톤 분석 (LLM 활용)
  - FMP API에서 earnings call transcript 수집 (기존 API에 포함 여부 확인 필요)
  - 전분기 대비 톤 변화를 LLM으로 분류:
    CAUTIOUS → NEUTRAL → CONFIDENT → AGGRESSIVE
  - Phase 2 초입 후보에 대해 톤 상승이 동반되는지 교차 확인

Phase 2: 내부자 매매 데이터 (추가 데이터 소스 필요)
  - SEC Form 4 데이터 (OpenInsider 등 무료 소스 존재)
  - Phase 1 후기 종목에서 내부자 매수 클러스터 감지
  - insider_signals 테이블에 기록

톤 분류 기준:
  - 가이던스 상향/하향
  - "acceleration", "inflection", "headwind" 등 키워드 빈도 변화
  - 질의응답에서 경영진의 구체성 수준 변화
```

### 난이도: 중간~높음 | 예상 3~5일

컨퍼런스콜 톤 분석은 기존 LLM 인프라 활용 가능. 내부자 데이터는 새 API 소스 필요.

## 제안 6: 섹터 간 시차(Lag) 패턴 축적

### 근거

섹터는 독립적으로 움직이지 않는다. "반도체가 Phase 2에 진입한 후 평균 N주 후에 반도체 장비가 Phase 2에 진입한다"는 식의 시차가 존재한다. 이 패턴은 서사와 무관하게 **순수 데이터에서 도출 가능**하며, 추가 데이터 소스 없이 기존 Phase + RS 데이터만으로 구현 가능.

### 구현 방안

```
sector_lag_patterns 테이블:
  id,
  leader_sector     -- "Semiconductors"
  follower_sector   -- "Semiconductor Equipment"
  avg_lag_weeks     -- 4.2
  sample_count      -- 12 (관측 횟수)
  confidence        -- 0.78 (상관계수 또는 p-value)
  last_updated      -- 2026-03-08

계산 로직:
  1. 각 섹터의 Phase 2 진입 시점을 시계열로 기록
  2. 섹터 쌍별로 Phase 2 진입 시차를 계산
  3. 충분한 관측(N ≥ 5) 후 평균 시차 + 신뢰도 산출
  4. 리더 섹터가 Phase 2 진입 시 → "N주 후 팔로워 섹터 주시" 알림

주간 리포트에 반영:
  "이번 주 Semiconductors Phase 2 진입.
   과거 패턴: Semiconductor Equipment가 평균 4.2주 후 Phase 2 진입 (12회 관측, 신뢰도 0.78)"
```

### 난이도: 중간 | 예상 2~3일

기존 Phase 데이터 + 섹터 분류 인프라 활용. 핵심은 충분한 데이터 축적 필요 (3개월+).

## 제안 7: 실패 패턴 체계적 축적

### 근거

현재 시스템은 "맞은 것"에서 학습하는 데 초점. 그러나 **"틀린 것"의 공통점**이 더 가치 있을 수 있다. 성공 패턴보다 실패 패턴이 더 일관되는 경향이 있어 학습 효율이 높다.

### 구현 방안

```
failure_patterns 테이블:
  id, created_at,
  pattern_name     -- "브레드스 악화 중 Phase 2 신호"
  conditions       -- {
                        market_breadth: "declining",
                        sector_rs: "rising_isolated",  -- 인접 섹터 미동반
                        volume_confirmed: false
                      }
  failure_count    -- 8
  total_count      -- 11
  failure_rate     -- 72.7%
  significance     -- p-value (이항 검정)
  
학습 흐름:
  1. Phase 2 신호 후 실패한 종목 수집 (Phase 1 회귀 또는 Phase 4 전환)
  2. 실패 시점의 시장 조건을 자동 기록:
     - 시장 브레드스 방향
     - 해당 섹터 RS 상승 시 인접 섹터 동반 여부
     - 거래량 확인 유무
     - 시장 레짐 (강세/약세)
     - 펀더멘탈 등급 (SEPA)
  3. 조건 조합별 실패율 산출
  4. 실패율 70%+ 패턴 → 필터링 규칙으로 승격

리포트 반영:
  "XYZ Phase 2 진입 신호 감지. 
   단, 현재 조건(브레드스 악화 + 섹터 고립 상승)에서 과거 실패율 72.7%.
   주의 필요."
```

### 난이도: 중간 | 예상 2~3일

기존 recommendations + Phase 변화 데이터 + 시장 스냅샷 결합. 위양성 추적(제안 2)과 시너지.

## 제안 8: 시장 레짐 분류기

### 근거

"어떤 종목이 좋은가"보다 **"지금이 종목을 찾을 때인가"**가 더 중요한 판단. 약세장에서 무리하게 Phase 2를 찾지 않는 것이, 강세장에서 하나 더 찾는 것보다 수익에 기여한다.

### 구현 방안

```
market_regime 분류:
  EARLY_BULL   -- 브레드스 반전, Phase 2 비율 상승 초기
  MID_BULL     -- 다수 섹터 Phase 2, RS 분산 확대
  LATE_BULL    -- 소수 종목 주도, 브레드스 피크 후 하락
  EARLY_BEAR   -- 브레드스 급락, Phase 4 비율 상승
  BEAR         -- 다수 섹터 Phase 4, 방어 섹터만 RS 상승

분류 입력:
  - 시장 브레드스 (Phase 2 비율 전체)
  - 브레드스 변화율 (5일/20일)
  - 선행/후행 지표 비율
  - 금리 환경 (Fed 정책 방향)
  - VIX 수준 및 변화율
  - 섹터 RS 분산도

레짐별 시스템 행동 조정:
  EARLY_BULL: 공격적 스캔, Phase 1 후기 종목 적극 포착
  MID_BULL: 정상 운영
  LATE_BULL: 추천 수 제한, 위양성 경고 강화
  EARLY_BEAR: 신규 추천 중단, 기존 추천 이탈 모니터링 집중
  BEAR: Phase 2 스캔 최소화, 바닥 신호 감시 모드

성과 추적:
  각 레짐에서의 추천 적중률을 분리 추적 → 
  "LATE_BULL에서 적중률 22%, MID_BULL에서 58%" 같은 메타 학습 축적
```

### 난이도: 중간 | 예상 2~3일

기존 브레드스 + RS 데이터 활용. 초기에는 규칙 기반, 데이터 축적 후 임계값 조정.

## 제안 9: 애널리스트 합의도(Consensus Degree) 추적

### 근거

4명의 토론 에이전트의 합의 정도 자체가 신호. 직관적으로는 만장일치가 좋을 것 같지만, **만장일치는 이미 시장에 반영된 컨센서스**일 가능성이 있다. 약간의 불일치가 있는 thesis가 더 높은 알파를 낼 수 있다.

### 구현 방안

```
thesis 테이블에 필드 추가:
  consensus_score  -- 4/4, 3/4, 2/4 (동의한 애널리스트 수)
  dissent_reason   -- 반대 의견 요약

추적:
  consensus_score별 thesis 적중률을 분리 추적
  - 4/4 만장일치 적중률: ?%
  - 3/4 다수 동의 적중률: ?%
  - 2/4 분열 적중률: ?%

기대 가설:
  만약 3/4가 4/4보다 적중률이 높다면 →
  "아직 컨센서스가 되지 않은 thesis"가 더 높은 알파를 가진다는 증거.
  이 경우 시스템은 만장일치 thesis보다 다수 동의 thesis를 우선시해야 함.
```

### 난이도: 낮음 | 예상 1일

토론 로그에서 동의/반대를 이미 추출 가능. 필드 추가 + 집계 로직만 필요.

## 제안 10: 시스템 메타 인사이트 수집

### 근거

6개월 운영 후 답할 수 있어야 하는 핵심 질문들이 있다. 이 메타 데이터 자체가 시스템의 방향을 결정하는 가장 중요한 피드백.

### 추적할 메타 질문

```
meta_insights:

Q1. LLM이 병목을 식별한 시점 vs 시장이 가격에 반영한 시점의 평균 시차는?
  → 양수(LLM이 먼저) = 시스템 엣지 실증
  → 0 또는 음수 = 서사 레이어의 가치는 스크리닝 자동화에 있음

Q2. 어떤 카테고리의 thesis가 가장 높은 적중률을 보이는가?
  → structural_narrative vs sector_rotation vs short_term_outlook
  → 가장 높은 카테고리에 토론 시간 배분을 늘림

Q3. 어떤 애널리스트 페르소나가 가장 정확한가?
  → 매크로 vs 테크 vs 지정학 vs 심리
  → 정확도가 낮은 페르소나 프롬프트 개선 또는 교체 근거

Q4. 교집합 필터의 실제 리프트(lift)는 얼마인가?
  → 기술적 단독 적중률 vs 서사 교집합 적중률
  → 리프트가 유의미하지 않으면 서사 레이어 재설계

Q5. 시스템이 가장 크게 틀린 사례의 공통점은?
  → worst 10 cases 분석
  → 특정 시장 조건, 특정 섹터, 특정 thesis 유형에 집중되는지
```

### 구현 방안

```
월간 메타 리포트 자동 생성:
  1. 위 질문들에 대한 데이터를 DB에서 추출
  2. LLM이 메타 분석 리포트 생성
  3. 결과에 따라 시스템 파라미터 조정 제안

meta_report 테이블:
  id, month,
  lead_time_avg    -- Q1 답변 (일 단위)
  best_category    -- Q2 답변
  best_analyst     -- Q3 답변
  intersection_lift -- Q4 답변 (%)
  worst_case_pattern -- Q5 답변 (텍스트)
```

### 난이도: 중간 | 예상 2~3일

대부분 기존 데이터의 집계/분석. 월간 자동 실행 스크립트 + LLM 리포트 생성.

## 리스크

- **LLM의 병목 추론 한계**: LLM은 학습 데이터에 있는 공급망 정보를 잘 매핑하지만, 아직 아무도 글로 쓰지 않은 새로운 병목은 도출하지 못할 수 있다. 기대치를 "새로운 발견"이 아니라 "알려진 구조의 체계적 정리"로 설정해야 함.
- **서사 thesis 검증 지연**: 구조적 서사는 검증에 2~3개월 소요. 단기에 학습 데이터가 쌓이지 않음.
- **복잡도 증가**: 12개 제안을 모두 구현하면 시스템 복잡도가 크게 증가. Wave 단위로 검증하며 진행 필수.
- **비용**: 토론 복잡도 증가 + 홀드아웃 + 메타 리포트로 API 비용 ~50% 증가 예상.
- **과최적화 리스크**: 필터를 너무 많이 쌓으면 신호가 0이 될 수 있음. 각 필터의 독립적 기여도를 메타 리포트(제안 10)에서 검증해야 함.
- **시장 레짐 분류 오류**: 레짐 분류가 틀리면 오히려 기회를 놓칠 수 있음. 초기에는 참고 정보로만 사용, 자동 행동 조정은 충분한 검증 후.
